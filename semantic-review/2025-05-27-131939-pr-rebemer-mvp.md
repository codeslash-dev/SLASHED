# reBEMer: subtree-scoped BEM class manager for Bricks structure panel

An MVP that injects "BEM" badges into the Bricks structure panel, opening a modal where users name a block + descendant elements in one pass and apply BEM classes as global classes to the subtree. The implementation is imperative-shell-around-Svelte: `main.js` owns the MutationObserver lifecycle and badge mounting, Svelte components own the panel UI, and `bricks-api.js` concentrates all Vue state access behind a single seam.

**Watch for:** No snapshot/rollback exists despite the design doc promising transactional apply — a mid-loop failure leaves partial mutations with no undo path (confirmed). The `mark_as_module` regex in the PHP enqueue can inject a second `type` attribute into the script tag on some WordPress output filters (likely). CSS custom properties on `:root` will collide with any other plugin using the same variable names (confirmed).

## High-level view

The Bricks-API seam (`bricks-api.js`) correctly exploits the flat-array-per-area shape of Bricks' state, and the splice-in-place pattern for `_cssGlobalClasses` preserves Vue reactivity without the fake-mutation hacks seen in prior plugins. The gap is that `upsertGlobalClass` pushes to the array — whether Vue's reactive intercept fires depends on Bricks' `globalClasses` being the original reactive reference (likely, but untested in this diff).

The apply loop (`apply.js`) processes rows sequentially with no try/catch boundary and no snapshot. If `upsertGlobalClass` throws partway through (it will when `_state` is null after a hot-reload), some elements will have their classes changed and others won't. The design doc describes a full snapshot/rollback protocol — it's absent from the implementation.

The PHP enqueue class is correctly gated behind both `bricks_is_builder_main()` and a capability check. The `mark_as_module` filter uses regex replacement that can produce `<script type="module" type="text/javascript"...>` if WordPress has already set the type attribute.

The CSS scoping strategy uses `.rebemer-*` class prefixes but leaks custom properties onto `:root`. The panel's `z-index: 100000` is undocumented relative to Bricks' own z-index stack.

Badge injection and reaping via MutationObserver is well-structured: single AbortController, debounce at 50ms, disconnected-host reaping. The gap is a stale 800ms auto-close timer that can race with a new panel mount on rapid re-clicks.

## Transactional safety of `applyToSubtree`

The design doc (section 10) specifies a snapshot before mutations, validation of each operation against live state, and rollback on throw. The actual implementation is a bare for-loop:

```js
for (const row of rows) {
  if (!row.include) continue;
  const el = api.findElement(row.id);
  if (!el) continue;
  // ... mutations ...
  api.setElementClasses(row.id, nextIds);
  count++;
}
```

If `upsertGlobalClass` throws (it explicitly `throw new Error('rebemer: not ready')` when `_state` is null — which can happen if Bricks hot-reloads and the probe becomes stale), the loop exits with some elements already mutated. The user sees a partial rename with no way to revert except browser-level undo (which Bricks may or may not support for programmatic state changes).

The fix: clone the affected `_cssGlobalClasses` arrays and labels before the loop, catch any exception, and restore from the clones.

## Bricks-API reactivity contract

`upsertGlobalClass` appends to `_state.globalClasses` via `list.push(...)`. Vue 3's reactive system intercepts array methods on reactive arrays, so this should trigger Bricks' watchers — but only if `_state.globalClasses` is the original reactive reference (not a destructured copy). The code preserves the reference chain (`_state = candidate` where `candidate` comes from `app?.__vue_app__?.config?.globalProperties?.$_state`), so the intercept *should* fire. If it doesn't, the symptom is that Bricks' class panel never sees the new classes until the next page load — a confusing silent failure. Using `list.splice(list.length, 0, entry)` would be a zero-cost defensive measure.

## PHP enqueue: `mark_as_module` regex

```php
return preg_replace( '/<script(\b[^>]*)>/', '<script type="module"$1>', $tag, 1 );
```

This inserts `type="module"` before existing attributes. If WordPress outputs `<script type="text/javascript" src="...">` (default on WP < 6.3 or with certain optimization plugins), the result is `<script type="module" type="text/javascript" src="...">` — invalid HTML. Browsers use the first `type` attribute per spec so it technically works, but security scanners flag it and it's trivially fixable:

```php
$tag = preg_replace('/\stype=["\'][^"\']*["\']/', '', $tag, 1);
return str_replace('<script ', '<script type="module" ', $tag);
```

## CSS `:root` variable leak

```css
:root {
  --rebemer-bg: #161a1d;
  --rebemer-fg: #e1e1e1;
  /* ... 11 more */
}
```

Another plugin or a future Bricks update could define `--rebemer-accent` and silently restyle the panel. Scoping under `.rebemer-panel` or `#slashed-rebemer-host` eliminates this with no functional change — all consumers are descendants of the panel anyway.

## Badge lifecycle and observer hygiene

`beforeunload` does not fire on SPA-style navigations within the Bricks builder (e.g., multi-page editing mode). If Bricks destroys and rebuilds the `[data-v-app]` container without a full page unload, `_state` goes stale but observers keep running. The probed state would fail on next mutation, but the observers never disconnect. Defensive fix: re-probe inside the MutationObserver callback; if `probe()` returns false, disconnect and surface a notice.

## Panel race condition on rapid re-clicks

`openPanel` calls `closePanel()` synchronously before mounting a new panel — safe because Svelte 5's `unmount` is synchronous. But `BemPanel.apply()` schedules `setTimeout(() => onClose?.(), 800)` which captures the `onClose` prop. If the user applies then clicks another badge within 800ms: old panel unmounts, new panel mounts, stale timeout fires `onClose` (same `closePanel` reference), closing the *new* panel unexpectedly.

Fix: track the timeout in a component-level variable and clear it on unmount via an `$effect` cleanup or `onDestroy`.

## Silent failure: non-Latin slugification

`slugify` strips all non-ASCII after NFKD normalization. For CJK, Arabic, or Cyrillic element labels, the result is `''`. The row is silently excluded from the apply with no error shown to the user. At minimum, surface a toast warning when one or more included rows produced an empty slug.

## Silent failure: "Applied to 0 elements"

If Bricks undo removes all subtree elements between panel open and apply, or if `_state` goes stale, every `findElement` returns null and the count stays 0. The success path returns `{ ok: true, count: 0 }` — the user sees "Applied to 0 elements" with a success toast. This should be treated as an error condition or at least a warning.
